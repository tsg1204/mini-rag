The Feature: Parsing Calories and Exercise Burn

Imagine your app lets users type something like:

“I ate a few old fashioned donuts today and sat in my rocking chair. I rocked vigorously! I imagine I burned around 3k calories overall”

and it calculates the net calories for the day.

That means your system needs to:

  1. Parse numbers like 3k to 3000.
  2. Distinguish between calories consumed and calories burned.
  3. Add or subtract them for the total.

Seems straightforward… until you try to test it.

What Most People Do (and Why It Sucks)

Seriously, I’ve seen these kinds of tests in a codebase and I am genuinely curious why they exist.

Here’s what most developers try first:

// ❌ calorieAgent.test.ts
jest.mock("@/lib/openai", () => ({
  chat: jest.fn(() => ({
    output: "You consumed 3000 and burned 750. Total: 2250!"
  }))
}));

import { calorieAgent } from "../calorieAgent";

test("calculates total calories", async () => {
  const result = await calorieAgent("I ate 3k calories but burned 750");
  expect(result).toBe("You consumed 3000 and burned 750. Total: 2250!");
});

This passes.

Every time.

Forever.

Because you’re not testing the real thing, you’re testing your mocked imagination of the model.

The day OpenAI changes its weights, or your teammate tweaks a prompt, or “3k” starts getting parsed as 3 instead of 3000…

Your mock won’t catch it.

You absolute fool.

You’ll deploy confidently, and production will break quietly.

Mocks give you a false sense of control in a non-deterministic system.

BigBax™️ can’t afford this kind of oversight.

Why Mocks Fail for LLMs

LLMs are non-deterministic.

  1. You can’t rely on the same text twice.
  2. Mocking assumes stability that doesn’t exist.
  3. They change over time.
  4. Model updates, temperature shifts, and prompt changes all alter output behavior.

Mocks freeze drift.

  • Your app evolves, your model evolves, but your mock stays stuck in time — hiding regressions instead of surfacing them.

They don’t validate structure.

  • You can’t test whether your code correctly parses or reacts to real responses if your mock is hard-coded text.

So if mocks are out then what the hell do we do?

The manager is getting ancy.

The Fix: Test Structure, Not (just) Words

Instead of asserting on raw strings, test what the model returns, not (just) how it says it.

The key is to:

  1. Force the model to return structured output (e.g. JSON).
  2. Validate that structure with a schema.
  3. Handle logic and math deterministically outside the model.

Let’s rebuild our calorie tracker the right way.

Step 1: The Agent (LLM Parser)

// calorieAgent.ts
import OpenAI from "openai";
import { z } from "zod";

const client = new OpenAI({ apiKey: process.env.OPENAI_KEY });

const CalorieSchema = z.object({
  consumedCalories: z.array(z.number()).describe('Best guess on calories consumed'),
  burnedCalories: z.array(z.number()).describe('Best guess on calories burned via normal movement and exercise')
});

export async function calorieAgent(userInput: string) {
  const result = await client.responses.parse({
    model: "gpt-4o-mini",
    input: `
    Extract calorie data from the user's text.
    Return two arrays:
    - consumedCalories: numbers related to eating
    - burnedCalories: numbers related to workouts and being an adult human
    Interpret "k" as 1000 (e.g., "3k" → 3000).
    If no calories mentioned, do your best to guess the calories consumed from their self-reporting.
    Return valid JSON only.
    `,
    text: { format: "json", schema: CalorieSchema },
  });

  return result.output_parsed;
}

Now the LLM doesn’t return free-form text — it returns predictable structure that our zod schema enforces

Step 2: The Utility Function

Keep the math outside the model so you can test it deterministically.

// calorieUtils.ts
export function calculateNetCalories(parsed: {
  consumedCalories: number[];
  burnedCalories: number[];
}) {
  const totalConsumed = parsed.consumedCalories.reduce((a, b) => a + b, 0);
  const totalBurned = parsed.burnedCalories.reduce((a, b) => a + b, 0);
  return totalConsumed - totalBurned;
}

Step 3: Add Useful Tests

Now that your agent returns consistent structure, you can write meaningful tests.

// calorieAgent.test.ts
import { calorieAgent } from "../calorieAgent";
import { calculateNetCalories } from "../calorieUtils";

describe("Calorie Agent", () => {

  it("parses and calculates correctly", async () => {
    const parsed = await calorieAgent(
      "I ate 3k calories today but burned 500 running and 250 lifting"
    );

    const total = calculateNetCalories(parsed);

    expect(parsed.consumedCalories).toContain(3000);
    expect(parsed.burnedCalories).toEqual(expect.arrayContaining([500, 250]));
    expect(total).toBe(2250);
  }, 30000);

  it("handles nonsense gracefully", async () => {
    const parsed = await calorieAgent("asdfghjkl qwertyuiop");
    const total = calculateNetCalories(parsed);

    expect(parsed.consumedCalories).toHaveLength(0);
    expect(parsed.burnedCalories).toHaveLength(0);
    expect(total).toBe(0);

  }, 30000);
});

Notice what we’re testing:

  • ✅ Structure (arrays exist and contain numbers)
  • ✅ Logic (our math function returns consistent totals)
  • ✅ Sanity (nonsense input produces empty arrays)

No mocks.

No false confidence.

Just real validation.

Step 4: Run Them Wisely

These tests hit the real API. That costs money.

Don’t run them in your normal CI suite.

Instead:

npm run test:ai

or filter by name:

jest --testNamePattern="Calorie Agent"

Run them after prompt updates, before deploys, or on a nightly schedule.

The Takeaways

  • ❌ Don’t mock LLM outputs — they’ll lie to you.
  • ✅ Force structured responses with schemas (Zod and/or JSON schema).
  • ✅ Keep logic and math deterministic.
  • ✅ Test behavior and structure, not phrasing.
  • ✅ Run tests selectively to control cost.

Mocks make you feel safe.

Real AI tests make you informed.

And the goal isn’t to prove your model’s perfect, it’s to know the exact moment it stops being predictable.
